# 사주로(sajuro.ai) 앱 UI/UX 개편안  
## 피그마용 컴포넌트 네이밍 규칙 + Props 정의표

---

## 1. 문서 목적

본 문서는 사주로(sajuro.ai) 앱의 UI/UX 개편안을 실제 피그마 컴포넌트 시스템에 바로 적용할 수 있도록 정리한 설계 기준서이다.  
주요 목적은 다음과 같다.

- 피그마에서 일관된 컴포넌트 네이밍 체계를 수립
- 공통 Variant 및 Props 구조를 정의
- 화면 조립 시 재사용 가능한 컴포넌트 패밀리 정리
- 개발 handoff 시 혼선을 줄일 수 있도록 필드 구조를 표준화

---

## 2. 네이밍 규칙 원칙

사주로 앱은 향후 분석 메뉴, 보관소, 기록, 궁합, 질문/답변, 이미지 분석 등으로 계속 확장될 가능성이 높다.  
따라서 피그마 컴포넌트는 반드시 **카테고리 / 역할 / 타입 / 상태** 기준으로 이름을 통일한다.

### 2.1 기본 네이밍 포맷

권장 포맷은 아래와 같다.

`Category / Component / Type / Variant`

예시:

- `Navigation / Header / Basic / Back`
- `Content / Card / Analysis / Default`
- `Input / Button / Primary / Large`
- `Overlay / BottomSheet / Selector / Default`

### 2.2 카테고리 분류

대분류는 아래와 같이 고정한다.

- `Navigation`
- `Content`
- `Input`
- `Feedback`
- `Overlay`
- `List`
- `Form`
- `Layout`
- `Icon`

### 2.3 상태값 표기 규칙

상태는 마지막에 붙인다.

예시:

- `Content / Card / Analysis / Default`
- `Content / Card / Analysis / Pressed`
- `Input / Button / Primary / Disabled`

상태 suffix 권장값:

- `Default`
- `Hover`
- `Pressed`
- `Selected`
- `Disabled`
- `Loading`
- `Empty`
- `Expanded`
- `Collapsed`

### 2.4 다크모드/라이트모드 관리 원칙

사주로는 다크 테마 중심이므로 기본은 Dark를 기준으로 설계하되, 이름에 직접 넣지 않고 Variant Prop으로 관리한다.

예시:

- `Theme = Dark`
- `Theme = Light`

즉, 이름에 Dark를 붙이지 말고 컴포넌트 Variant에서 분기하는 방식을 권장한다.

## 3. 피그마 Variant 공통 규칙

### 3.1 공통 Variant Key 권장

모든 컴포넌트에 아래 key 체계를 최대한 통일한다.

- `Type`
- `Size`
- `State`
- `Theme`
- `Icon`
- `Badge`
- `Action`
- `Divider`
- `Selected`

예시:

- `Type = Analysis / Record / Profile`
- `Size = S / M / L`
- `State = Default / Pressed / Disabled / Loading`
- `Theme = Dark / Light`
- `Icon = On / Off`
- `Badge = None / One / Two`
- `Action = None / One / Two / Three`

### 3.2 Boolean Props 최소화 원칙

피그마에서 Boolean이 너무 많으면 관리가 복잡해진다.  
가능하면 아래처럼 enum형으로 관리한다.

나쁜 예:

- `HasIcon = true/false`
- `HasBadge = true/false`
- `HasButton = true/false`

좋은 예:

- `Icon = None / Leading / Trailing`
- `Badge = None / One / Two`
- `CTA = None / Single / Double`

## 10. 팀 운영 규칙

### 규칙 1
새 컴포넌트를 만들기 전에 기존 카테고리 안에서 확장 가능한지 먼저 검토한다.

### 규칙 2
텍스트 변경만 필요한 경우는 새 Variant를 만들지 말고 text override로 처리한다.

### 규칙 3
상태 차이만 있을 때는 별도 컴포넌트 복제가 아니라 State Variant를 추가한다.

### 규칙 4
카드 내부 구조가 70% 이상 같으면 하나의 컴포넌트 패밀리로 유지한다.

### 규칙 5
화면 단위 프레임명도 규칙화한다.

예시:

- `Screen / Home / Default`
- `Screen / Analysis / Grid`
- `Screen / Vault / List`
- `Screen / Result / Wealth`
- `Screen / QA / Thread`

## 12. 결론

피그마에서 바로 쓰기 위한 핵심 원칙은 아래 3가지이다.

- 이름은 **카테고리 / 역할 / 타입** 기준으로 고정한다.
- Props는 **Type, Size, State, Theme** 중심으로 단순화한다.
- 카드 / 버튼 / 배지 / 시트를 공통화하여 화면마다 재사용한다.

사주로는 앞으로 분석 메뉴, 기록, 궁합, 질문/답변, 이미지 분석까지 계속 확장될 가능성이 높으므로,  
지금부터라도 **분석 카드 / 명식 카드 / 기록 카드**를 중심으로 시스템을 잡아두는 것이 가장 중요하다.
